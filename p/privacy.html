How this site handles privacy.

Last updated: 22 May 2026. This policy explains how mas0ng.com handles personal data across the public pages, login pages, private dashboard pages, and Cloudflare Workers that route, protect, and run parts of the site.

This page is written to follow the main transparency requirements expected under UK data protection law, including the UK GDPR, the Data Protection Act 2018, and PECR rules for cookies and similar technologies.

No public tracking scripts The public pages do not load third-party tracking or analytics scripts.
Protected areas use sessions Auth cookies and session checks are used to confirm access to private pages.
Workers run private logic Cloudflare Workers protect dashboard pages and keep private service keys server-side.

Who is responsible for this site

The site owner and controller for mas0ng.com is Mason. For privacy questions, requests, or concerns, contact mason@mas0ng.com.

This is a personal website with private dashboard features. It is not intended to collect large amounts of public personal data.

No Data Protection Officer is currently appointed because this site is not intended to carry out large-scale, high-risk, or public-authority processing. Privacy requests should be sent to the contact email above.

What this policy covers

This policy applies to pages and services under mas0ng.com. That includes the public profile page, the privacy page, the login flow, the dashboard, and API routes handled by Cloudflare Workers.

Some public links may take you to external websites, such as social platforms. Those services have their own privacy policies and are responsible for how they handle data after you leave mas0ng.com.

This policy explains how the site is intended to operate. It does not override legal rights that you may have under UK data protection law, and it does not create additional contractual rights beyond those required by law.

Important limits

The public parts of mas0ng.com are provided for basic information and links. The private dashboard is intended for authorised use only.

  • Do not send sensitive personal information through this site unless specifically requested.
  • Do not attempt to access private dashboard routes unless you are authorised to do so.
  • Do not rely on public pages as a secure messaging system.
  • If you follow an external link, you leave mas0ng.com and the external site becomes responsible for its own processing.

Where information comes from

Most information is collected directly from your browser or from actions you take on the site. For example, a page request is created when you load a page, a session is created when a login succeeds, and a dashboard action is created when an authorised user submits a dashboard request.

Some operational information may also be generated automatically by Cloudflare or browser/server infrastructure, such as request timestamps, response codes, and security events.

Why information is used

Information is only intended to be used for clear site functions. The main purposes are:

  • to load and display public pages;
  • to remember the privacy notice choice in the browser;
  • to show a warning before opening external links;
  • to verify logins and protect private dashboard pages;
  • to process private dashboard and worker API requests;
  • to keep secrets and service keys away from public website files;
  • to prevent unauthorised access, abuse, or broken requests;
  • to debug errors and keep the site working.

Lawful bases under UK GDPR

Where UK GDPR applies, the lawful basis depends on the purpose:

  • Legitimate interests: running the site, protecting private pages, preventing misuse, keeping server-side services secure, and diagnosing technical issues.
  • Contract or steps before a contract: where a private feature is specifically requested and needs to operate as part of providing that feature.
  • Legal obligation: where information must be kept or shared to comply with applicable law.
  • Consent: where consent is required for non-essential cookies or similar technologies. The current public pages do not use third-party tracking scripts.

Where legitimate interests are relied on, the interests are running and protecting the site, preventing unauthorised access, avoiding exposure of private keys, handling technical errors, and keeping the private dashboard available to authorised users. These interests are balanced against the limited nature of the data handled and the separation between public pages and private worker-side logic.

Login and authentication

The site has a private login system for protected areas. When login succeeds, the auth service creates a session so the site can recognise that the browser is allowed to access private pages.

Authentication may use a username, password verification, one-time code checks, session tokens, and browser cookies. Passwords and secrets should not be exposed to public pages or committed into public website files.

Session tokens are intended to be temporary. If a session expires, is deleted, or fails verification, private access should require logging in again.

Failed login attempts, expired sessions, invalid tokens, and suspicious requests may be rejected automatically. This is necessary to keep private areas protected.

Access control and authorisation

Protected dashboard pages and private API routes should only be available after a valid session has been verified. A successful login does not grant access to external accounts or services unless those services have separately been configured for the dashboard.

Worker-side checks are intended to prevent public visitors from calling private APIs directly. If a check fails, the request may be redirected, rejected, or return an error.

Authorisation rules may be changed at any time to improve security, remove features, or respond to suspected misuse.

Cloudflare Workers

Parts of mas0ng.com are handled by Cloudflare Workers. Workers are server-side services that can receive a request, check whether it is allowed, and return the right page or response.

Auth worker

The auth worker handles login, logout, session creation, and session verification. Other protected parts of the site can ask the auth worker whether the current visitor is logged in.

Dashboard worker

The dashboard worker protects dashboard pages and dashboard API routes. It should verify the session before returning private content or running private actions.

Server-side API requests

Some dashboard features may call other services from the worker side. This helps keep private keys or service tokens away from the browser. The browser should only receive the result needed for the page to work.

Worker logs

Cloudflare and worker tooling may process request metadata, error details, and security events. This can include IP address, request path, timestamp, user agent, and response status.

Worker code may also add server-side secrets to outbound requests where a private dashboard feature requires them. Those secrets should not be returned to the browser unless the feature specifically requires a safe public value.

Private dashboard features

Private dashboard features may send requests to worker routes to show statuses, run actions, or connect to services configured for the dashboard.

Where private services need keys or tokens, those values should stay server-side in Worker environment variables, bindings, or other private configuration. They should not be placed into public static website files.

If a dashboard action sends data to an external service, that service may process the data according to its own privacy terms.

Security

The site is designed so private pages and API actions require a valid session. Workers should check authorisation before serving protected dashboard content or running private API calls.

Private service tokens, admin tokens, and API keys should be stored server-side and should not be exposed through public pages. The site separates public static files from private worker-side logic to reduce the risk of leaking secrets.

No security system is perfect. If you think something is wrong, contact mason@mas0ng.com.

If a personal data breach is suspected, reasonable steps should be taken to investigate, reduce harm, and notify affected people or the ICO where legally required.
